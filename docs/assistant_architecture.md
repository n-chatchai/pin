# Pin Assistant Architecture

## 1. Core Concept (แนวคิดหลัก)
เปลี่ยนระบบความสามารถของ AI จากการติดตั้งแยกชิ้น (Tools/Skills) เป็นการติดตั้งแบบ **"Use Case Package"** โดยใช้ชื่อเรียกฝั่งผู้ใช้ว่า **"ผู้ช่วย (Assistants)"** 

เมื่อผู้ใช้ติดตั้งผู้ช่วย 1 คน ระบบจะทำการจัดเตรียม **สมอง/ทักษะ (Capabilities)** และ **ท่อเชื่อมต่อ (Connectors)** ที่จำเป็นสำหรับผู้ช่วยคนนั้นให้โดยอัตโนมัติ 

---

## 2. Data Model (โครงสร้างฐานข้อมูล)
เพื่อให้ระบบยืดหยุ่นและรองรับการแชร์ทรัพยากร (เช่น ผู้ช่วยหลายคนใช้เครื่องมือเดียวกันได้) ฐานข้อมูลจึงถูกออกแบบในรูปแบบ **Many-to-Many (M:M)** ดังนี้:

- **`assistants`**: เก็บรายชื่อแพ็กเกจผู้ช่วย (เช่น `gmail_summarizer`, `tutor`)
- **`connectors`**: เก็บการตั้งค่า API หรือท่อเชื่อมต่อโลกภายนอก (เช่น `google_oauth`, `mcp_server`)
- **`capabilities`**: เก็บเครื่องมือ (Tools) หรือ ทักษะ (Skills) รวมถึงซับเอเจนต์ โดยสามารถโยงไปหา `connectors` ที่ต้องใช้ได้
- **`assistant_capabilities`**: ตารางเชื่อม (Junction Table) ที่ระบุว่าผู้ช่วยคนไหน มีสิทธิ์ใช้ Capability ตัวไหนบ้าง

*ข้อดี: หากลบผู้ช่วยออก ระบบจะลบแค่สิทธิ์ในตารางเชื่อม โดยไม่กระทบกับเครื่องมือหรือท่อที่ผู้ช่วยคนอื่นกำลังใช้งานอยู่*

---

## 3. Interaction Modes (รูปแบบการสื่อสารฝั่งแอป Flutter)
เพื่อให้แอปพลิเคชันรองรับ Use Case ที่หลากหลาย ในฟิลด์ `metadata_json` ของ Capability (กลุ่ม Subagent) จะมีการระบุ `interaction_mode` ไว้ เพื่อให้แอป Flutter รู้ว่าต้องจัดการกับผู้ช่วยคนนี้อย่างไร:

### 3.1. Delegation Mode (โหมดมอบหมายงาน)
- **พฤติกรรม:** บอทหลักเป็นผู้รับหน้าพูดคุยกับผู้ใช้ เมื่อเจองานเฉพาะทาง บอทหลักจะโยนงานให้ผู้ช่วย (Subagent) ไปทำแบบเงียบๆ ใน Sandbox (จำกัดสิทธิ์ให้ใช้ได้แค่เครื่องมือที่กำหนด) เมื่อเสร็จสิ้น ผู้ช่วยจะส่งผลสรุปกลับมาให้บอทหลักเพื่อนำไปตอบผู้ใช้
- **เหมาะสำหรับ:** ผู้ช่วยนักวิจัย (Researcher), ผู้ช่วยค้นหาสินค้า (Shopper)

### 3.2. Handoff Mode (โหมดส่งไม้ต่อ / เปลี่ยนสมอง)
- **พฤติกรรม:** บอทหลักส่งต่อห้องแชทให้ผู้ช่วยเข้ามารับช่วงต่อแบบ 100% (Switch Brain) ผู้ใช้จะคุยกับผู้ช่วยโดยตรงโดยไม่มีบอทหลักมาคั่นกลาง จนกว่าผู้ช่วยจะคืนสิทธิ์กลับไป
- **เหมาะสำหรับ:** ติวเตอร์สอนภาษา (Tutor), โหมดฝึกสัมภาษณ์งาน

*(หมายเหตุ: สำหรับโหมด Background/Trigger ที่คอยเฝ้าระวังและส่งแจ้งเตือน จะถูกแยกออกไปจัดการในฐานะ App Feature โดยไม่ปะปนกับระบบสนทนาของ Assistant)*

---

## 4. Backward Compatibility (การรองรับระบบเดิม)
เพื่อให้แอป Flutter เดิมสามารถทำงานต่อได้ทันทีโดยไม่ต้องแก้ไขโค้ด API `/catalog` ของ Backend จะทำหน้าที่ดึงข้อมูลจากตาราง `capabilities` ทั้งหมดที่เปิดใช้งานอยู่ แล้วนำมาจัดกลุ่ม (Group by) ตามประเภท `kind` กลับไปเป็น JSON Arrays 3 ก้อนเดิม ได้แก่ `tools`, `skills`, และ `subagents`
