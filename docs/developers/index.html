บทที่ 1

ภาพรวม & สถาปัตยกรรม

พัฒนา "ความสามารถ" (Capability) ให้กับปิ่น และเผยแพร่ลง Store เพื่อให้ผู้ใช้งานทุกคนสามารถเรียกใช้งานได้ โดยที่เซิร์ฟเวอร์ของปิ่นจะไม่มีวันเข้าถึงหรือบันทึกข้อมูลส่วนตัวของผู้ใช้

ปิ่นคืออะไร?

ปิ่น (Pin) คือผู้ช่วย AI ภาษาไทยบนโทรศัพท์มือถือ โดยมี "สมอง" (Local Engine) ทำงานอยู่บนอุปกรณ์ของผู้ใช้โดยตรง และเชื่อมต่อกับ Large Language Model (LLM) ผ่านทาง Proxy ที่ตาบอด (Blind Proxy) ซึ่งทำหน้าที่เพียงส่งต่อข้อมูลไปยังโมเดลเท่านั้น โดยไม่มีการเก็บบันทึก และไม่สามารถมองเห็นเนื้อหาบทสนทนาหรือตัวตนของผู้ใช้งานแต่อย่างใด

ความสามารถใหม่ ๆ จะถูกดึงมาจาก Catalog แบบเรียลไทม์ (Runtime) ทำให้นักพัฒนาสามารถเพิ่มหรืออัปเดตเครื่องมือได้ทันทีโดยไม่ต้องรอให้แอปพลิเคชันหลักอัปเดตเวอร์ชัน และนี่คือช่องทางที่คุณสามารถเข้ามาสร้างสรรค์บริการใหม่ ๆ ได้

ประเภทของความสามารถ (Capability)

ประเภทสิ่งที่นักพัฒนาต้องเตรียมกลไกการทำงาน
tool (Remote Tool)HTTP Server ของคุณProxy จะส่ง Parameter แคบ ๆ ไปยัง Endpoint ของคุณ และนำผลลัพธ์กลับมาแสดงผล
mcp (MCP Server)MCP Server URLProxy จะทำการเชื่อมต่อกับ MCP Server ของคุณ เพื่อเปิดชุดเครื่องมือ (Tools) หลาย ๆ ตัวพร้อมกัน
skillชุดคำสั่ง (Instructions) ข้อความคำสั่งที่ถูกฉีดเข้าไปใน Persona ของปิ่น (ไม่ต้องมี Server หรือ Code ใด ๆ)
subagentSystem Prompt + รายชื่อ Toolsผู้ช่วยย่อยที่ปิ่นจะสามารถมอบหมายงานเฉพาะทางให้ทำใน Sandbox

หมายเหตุ: เอกสารฉบับนี้จะเน้นไปที่การสร้าง Tool เป็นหลัก เนื่องจากพบได้บ่อยที่สุด สำหรับ MCP, Skill หรือ Subagent ก็ยังคงอ้างอิงมาตรฐานเดียวกันในเรื่องของ Parameters และการสร้างการ์ด (UI)

สถาปัตยกรรมระบบ

ผู้ใช้งานพิมพ์คำสั่ง: “ขอข่าวใหม่ๆ หน่อย” │ ▼ [Local Engine] บนเครื่อง ──── ดึงข้อมูล /catalog ────► รับทราบรายชื่อเครื่องมือ │ │ AI Model ตัดสินใจเรียกใช้: news_reporter(topic: "") ▼ [Blind Proxy] /tool/news_reporter ── ส่งต่อแค่ {topic:""} ──► [Endpoint ของคุณ] │ │ ดึง RSS, สรุปผล ◄────────────── ตอบกลับเป็น {flex: การ์ด} ─────────────┘ ▼ แอปพลิเคชันนำข้อมูลไปเรนเดอร์เป็นการ์ด UI ด้วยธีมของปิ่น (Native Component)

หลักการทำงานแบบ "ตาบอด" และ PII Gate

  • เครื่องมือจะได้รับข้อมูล เฉพาะคีย์ที่ประกาศไว้ใน argKeys เท่านั้น โดยแอปพลิเคชันจะทำการตัด Parameter อื่น ๆ ทิ้งทั้งหมดก่อนส่งออกจากเครื่อง
  • ห้ามรับคีย์ที่เกี่ยวข้องกับข้อมูลส่วนบุคคล (PII) โดยเด็ดขาด เช่น identity, conversation, email, token, name, phone, user_id, หรือ history ระบบรักษาความปลอดภัย (Audit) จะทำการ บล็อก (Block) ทันที
  • Logs ของ Proxy จะเก็บบันทึกเพียงแค่ Metadata (เช่น ชื่อเครื่องมือ, รายชื่อคีย์พารามิเตอร์, และสถานะการตอบกลับ) แต่จะไม่บันทึก Value ที่อยู่ข้างใน
🛡️
ทำไมระบบถึงต้อง "ตาบอด"?
ปิ่นให้คำมั่นสัญญากับผู้ใช้งานว่าข้อมูลส่วนตัวจะไม่มีวันรั่วไหล ดังนั้นเครื่องมือจากภายนอกจึงต้องถูกออกแบบมาให้ "ตาบอด" หากบริการของคุณมีความจำเป็นต้องอ้างอิงข้อมูลผู้ใช้ (เช่น ระบบสมาชิก) ให้พัฒนาในรูปแบบ mcp ซึ่งสามารถจดจำผู้ใช้ผ่าน end_user_ref ที่ Proxy สร้างให้ (ไม่ใช่ตัวตนจริงของผู้ใช้) แทน

รูปแบบผลลัพธ์ที่ตอบกลับได้

  • {"text": "..."} — ข้อความดิบ เพื่อให้โมเดลนำไปเรียบเรียงและสรุปความต่อ (เหมาะกับผลการค้นหาข้อมูล)
  • {"flex": { ... }} — การ์ดสำเร็จรูป ที่พร้อมแสดงผลให้ผู้ใช้เห็นทันที (ดูรายละเอียดในบทที่ 4)
บทที่ 2

เริ่มต้นใช้งาน

เริ่มต้นจากศูนย์จนสามารถสร้างเครื่องมือแรกของคุณได้ภายใน 5 นาที

1. เข้าสู่ระบบ Developer Portal

เข้าไปที่หน้า /developers และเข้าสู่ระบบด้วย บัญชีปิ่น (บัญชี Matrix ของคุณบน pin-chat) ที่พื้นที่นี้คุณจะสามารถส่ง Capability ใหม่เข้าสู่ระบบ, ติดตามสถานะการรีวิว, และจัดการเครื่องมือที่เผยแพร่ไปแล้วได้

2. เลือกประเภทของเครื่องมือ

หากคุณต้องการ...ควรเลือกใช้
ดึงข้อมูลหรือประมวลผลผ่านเซิร์ฟเวอร์ของคุณ แล้วตอบกลับเป็นการ์ดสวยงามtool
เชื่อมต่อกับชุดเครื่องมือมาตรฐานผ่านโปรโตคอล MCPmcp
กำหนดลักษณะนิสัย สไตล์การพูด หรือให้ความรู้เฉพาะทางแก่ปิ่น โดยไม่ต้องพึ่งพาเซิร์ฟเวอร์skill
มอบหมายงานเฉพาะทางที่มีความซับซ้อนให้กับผู้ช่วยย่อยทำงานแทนsubagent

3. Quickstart — โครงสร้าง Tool ขั้นต่ำ

เครื่องมือแบบ Remote Tool คือ Endpoint ธรรมดาที่รับ Request เป็น JSON และตอบกลับเป็น JSON คุณสามารถเลือกใช้เทคโนโลยีใดก็ได้ที่สะดวก ลองดูตัวอย่างการเขียนด้วย Python (FastAPI) หรือ Node.js (Express) ด้านล่าง:

# ตัวอย่าง FastAPI — Endpoint เดียวพร้อมใช้งาน
from fastapi import FastAPI, Request

app = FastAPI()

@app.post("/run")
async def run(req: Request):
    # รับเฉพาะคีย์ที่คุณได้ประกาศไว้ใน argKeys
    args = await req.json()
    place = (args.get("place") or "").strip()
    
    # ... ทำการดึงข้อมูลพยากรณ์อากาศของเมืองนั้น ...
    
    return {"flex": {
        "header": {"icon": "sun", "title": f"สภาพอากาศที่ {place}"},
        "body": [{"type": "bignum", "value": "32°C", "sub": "โอกาสเกิดฝน 20%"}],
    }}
// ตัวอย่าง Node.js Express — Endpoint เดียวพร้อมใช้งาน
const express = require('express');
const app = express();
app.use(express.json());

app.post('/run', (req, res) => {
    // รับเฉพาะคีย์ที่คุณได้ประกาศไว้ใน argKeys
    const args = req.body || {};
    const place = (args.place || "").trim();
    
    // ... ทำการดึงข้อมูลพยากรณ์อากาศของเมืองนั้น ...
    
    res.json({"flex": {
        "header": {"icon": "sun", "title": `สภาพอากาศที่ ${place}`},
        "body": [{"type": "bignum", "value": "32°C", "sub": "โอกาสเกิดฝน 20%"}],
    }});
});

app.listen(3000, () => console.log('Server is running'));

คุณสามารถ Deploy โค้ดนี้ไว้ที่ใดก็ได้ที่เชื่อมต่ออินเทอร์เน็ตได้และมี HTTPS รองรับ จากนั้นให้นำ URL ไปกรอกในระบบตอนส่ง Submit

4. ส่งเข้ารีวิว (Submit)

ที่หน้า Portal ให้กดปุ่ม Submit พร้อมกับแนบ Payload (JSON) สำหรับอธิบายเครื่องมือของคุณ (รายละเอียดฟิลด์ทั้งหมดดูได้ที่บทที่ 5) ตัวอย่าง Payload สั้น ๆ:

{
  "name": "get_weather",
  "kind": "remote",
  "description": "ดูพยากรณ์อากาศของเมืองที่ระบุ",
  "endpoint": "https://your-server.com/run",
  "parameters": {
    "type": "object",
    "properties": {
      "place": { "type": "string", "description": "ชื่อเมือง" }
    },
    "required": ["place"]
  },
  "argKeys": ["place"],
  "label": "พยากรณ์อากาศ",
  "category": "ข้อมูลทั่วไป",
  "provider": "YourCo"
}

ภาพรวมเส้นทางการพัฒนา (Lifecycle)

สร้าง Endpoint ─► Submit ใน Portal ─► ระบบ Auto-audit ตรวจสอบ ─► ทีมงานปิ่นรีวิว ─► ติดตั้งลง Catalog ─► เจ้าของกดเผยแพร่ ─► ใช้งานจริง (Live)
บทที่ 3

การสร้างเครื่องมือ · Tool & MCP

ความเข้าใจเรื่องข้อตกลง (Contract) ระหว่าง Endpoint ของคุณกับ Proxy ของปิ่น

ข้อตกลงการเชื่อมต่อ HTTP

  • Method: POST ไปยัง Endpoint ของคุณเสมอ
  • Input: JSON Body จะมีเฉพาะคีย์ที่คุณประกาศไว้ใน argKeys เท่านั้น (Proxy จะดักจับและลบข้อมูลส่วนเกินทิ้งทันที)
  • Output: ต้องตอบกลับเป็น JSON เท่านั้น ซึ่งสามารถตอบเป็นข้อความ {"text": ...} หรือการ์ด {"flex": ...}
  • Timeout: การประมวลผลควรจบภายใน ~30 วินาที หากเกิดข้อผิดพลาด ให้ตอบกลับเป็นข้อความแจ้งเตือนที่ผู้ใช้เข้าใจได้ เช่น {"text": "เกิดข้อผิดพลาดในการเชื่อมต่อเซิร์ฟเวอร์"}

การประกาศโครงสร้างเครื่องมือ (Declaration)

ฟิลด์ความหมายและการนำไปใช้งาน
nameชื่อทางเทคนิคของเครื่องมือ (ใช้ตัวอักษร a-z, _ ) ซึ่งโมเดลจะใช้เรียกคำสั่งนี้
descriptionคำอธิบายภาษาไทย เพื่อให้โมเดล AI เข้าใจอย่างชัดเจนว่า "ควรเรียกใช้เครื่องมือนี้ในสถานการณ์ใด"
parametersกำหนดโครงสร้างตัวแปรในรูปแบบ JSON-Schema เพื่อบังคับให้โมเดลส่งข้อมูลตาม Format ที่ระบุ
argKeysรายชื่อคีย์ที่อนุญาตให้ส่งผ่านได้ (PII Allowlist) โดยต้องตรงกับ properties ที่ประกาศไว้ด้านบน
endpointURL ของเซิร์ฟเวอร์คุณ (เฉพาะการสร้างแบบ Remote Tool)

กฎ "Minimal-arg" ขอเท่าที่จำเป็น

  • ควรออกแบบรับ Parameter ด้วย คำค้นหา (Query) ที่แคบที่สุด เท่าที่จำเป็นสำหรับการทำงาน (เช่น รับชื่อสถานที่ place หรือหัวข้อ topic)
  • ทุกคีย์ที่อยู่ใน argKeys จะต้องไม่บ่งบอกถึงข้อมูลส่วนบุคคลเด็ดขาด มิฉะนั้นระบบรักษาความปลอดภัยจะทำการ Block อัตโนมัติ
⚠️
รายการคีย์ต้องห้าม (Auto-blocked): identity, conversation, prefs, user, email, token, secret, password, name, phone, address, user_id, history

ตัวอย่างโครงสร้างโค้ดเซิร์ฟเวอร์เชิงลึก (News Reporter)

@app.post("/run")
async def run(req: Request):
    body = await req.json()
    # ดึงค่าหัวข้อข่าว ซึ่งเป็นคีย์เดียวที่ผ่าน argKeys มาได้
    topic = (body.get("topic") or "").strip()
    
    # สมมติฟังก์ชันดึง RSS Feeds และเลือกมา 6 ข่าวล่าสุด
    items = fetch_rss(topic)[:6]
    
    # สร้างการ์ด Flex Message
    cards = [{
        "header": {"title": item.title},
        "body":   [{"type": "text", "text": item.summary}],
        "footer": {
            "icon": "news", 
            "text": item.source,
            "trailing": "อ่านต่อ →", 
            "action": {"data": item.link}
        },
    } for item in items]
    
    # ตอบกลับเป็น Carousel ของการ์ด
    return {"flex": {"carousel": cards}}
app.post('/run', (req, res) => {
    // ดึงค่าหัวข้อข่าว ซึ่งเป็นคีย์เดียวที่ผ่าน argKeys มาได้
    const topic = (req.body.topic || "").trim();
    
    // สมมติฟังก์ชันดึง RSS Feeds และเลือกมา 6 ข่าวล่าสุด
    const items = fetchRss(topic).slice(0, 6);
    
    // สร้างการ์ด Flex Message
    const cards = items.map(item => ({
        "header": {"title": item.title},
        "body":   [{"type": "text", "text": item.summary}],
        "footer": {
            "icon": "news", 
            "text": item.source,
            "trailing": "อ่านต่อ →", 
            "action": {"data": item.link}
        }
    }));
    
    // ตอบกลับเป็น Carousel ของการ์ด
    res.json({"flex": {"carousel": cards}});
});

ทางเลือกสำหรับระบบที่ซับซ้อน: MCP Server

หากคุณมีระบบที่รองรับโปรโตคอล MCP (Model Context Protocol) อยู่แล้ว คุณสามารถส่งประเภทเป็น mcp ได้ โดยระบุ url, headers (เช่น ค่า Authentication สำหรับระบบคุณเอง) และรายการ tools ทั้งหมด (พร้อม name, description, parameters, argKeys)

Proxy ของปิ่นจะทำหน้าที่ครอบและเชื่อมต่อให้โดยอัตโนมัติ นอกจากนี้ฝั่ง Admin ของปิ่นยังสามารถกำหนด Defaults/Params ที่ Proxy จะฉีดเข้าไปทุกครั้งที่มีการเรียกใช้งานได้ (เช่น end_user_ref รหัสอ้างอิงผู้ใช้ที่ไม่ระบุตัวตน หรือ lang ภาษาที่ใช้) — ทั้งนี้ เครื่องมือของคุณจะยังคงไม่ทราบตัวตนที่แท้จริงของผู้ใช้งานอยู่ดีเพื่อรักษาความเป็นส่วนตัว

การจัดการภาษา

ข้อความต่างๆ ในผลลัพธ์หรือ UI Card ควรถูกจัดทำในภาษาเดียวกับที่ผู้ใช้ตั้งค่าไว้ (ส่วนใหญ่คือภาษาไทย หรือภาษาอังกฤษ) ยกเว้นชื่อเฉพาะทาง โค้ดโปรแกรม หรือ URL ที่ควรคงภาษาต้นฉบับไว้

บทที่ 4

การ์ดผลลัพธ์ / UI · Flex Spec

การออกแบบอินเทอร์เฟซ โดยส่งโครงสร้าง JSON (Flex) เพื่อให้แอปพลิเคชันเรนเดอร์ร่วมกับธีมหลักของปิ่นอย่างสวยงาม

🎨
นักพัฒนาคุมเนื้อหา · แอปพลิเคชันคุมสไตล์
เพื่อให้ประสบการณ์การใช้งาน (UX) ของปิ่นมีความเป็นหนึ่งเดียว (Brand Consistency) โทนสี, ฟอนต์, เงา, และระยะห่างต่างๆ จะถูกล็อกควบคุมโดยธีมของแอปทั้งหมด หากคุณพยายามส่ง Argument เพื่อบังคับสไตล์เอง ระบบจะละเว้นค่านั้นๆ

โครงสร้างระดับบนสุด (Top-level)

คุณสามารถตอบกลับด้วย {"flex": { ... }} โดยเลือกโครงสร้างได้สองแบบหลัก:

  • {"carousel": [card1, card2, ...]} = ชุดการ์ดเรียงต่อกันในแนวนอน เลื่อนปัดซ้ายขวาได้ (Swipeable) ควรมี 2 ใบขึ้นไป
  • {"header": ..., "body": ..., "footer": ...} = การ์ดเดี่ยว แสดงผลเต็มความกว้างของหน้าจอแชท

Header  ส่วนบนสุด

รับข้อมูล: { icon?, title, subtitle?, trailing? }

Footer  ส่วนล่างสุด

รับข้อมูล: { icon?, text, trailing?, action?: {data} }
หากใส่ฟิลด์ action.data แถบ Footer จะสามารถกดแตะได้ (รองรับการเปิดลิงก์ หรือส่ง Postback กลับไปยังระบบ)

Body = รายการบล็อกแสดงเนื้อหา

ฟิลด์ Body รับค่าเป็น Array ของบล็อก (Blocks) โดยแต่ละบล็อกจะแสดงผลเรียงต่อกันจากบนลงล่าง ชนิดของบล็อกที่รองรับมีดังนี้:

Type (ชนิด)Fields (ฟิลด์ที่รองรับ)หน้าที่และการแสดงผล
texttext, style?(body·title·muted), color?(pos·neg·accent·muted), maxLines?ข้อความบรรยายทั่วไป
kvk, v, sub?, color?แถวข้อมูลแบบ Key-Value (ป้ายชื่อซ้าย ค่าข้อมูลขวา)
bignumvalue, sub?, delta?ตัวเลขขนาดใหญ่โดดเด่น (หากกำหนด delta ระบบจะแสดงสีแดง/เขียวอัตโนมัติ)
tasktag, text, due?แสดงรายการงาน (Task) พร้อมป้ายกำกับและวันครบกำหนด
agendatime, textแสดงตารางเวลาและกิจกรรม
progressvalue(0–1), label?แสดงแถบแสดงความคืบหน้า (Progress Bar)
gaugevalue, max?, label?แสดงกราฟแบบเกจครึ่งวงกลม
barsvalues[], labels[], highlight?กราฟแท่งแบบง่าย
linepoints[]กราฟเส้นแนวโน้ม
divider—ขีดเส้นแบ่งคั่นระหว่างเนื้อหา
buttonlabel, action{data}ปุ่มกด เพื่อให้ผู้ใช้กด Action (Postback)
htmlhtmlทางเลือกสุดท้าย (รองรับแค่ HTML Subset เบื้องต้น)

รายชื่อไอคอน (สำหรับ Header/Footer)

คุณสามารถใส่คีย์ icon ด้วยชื่อต่อไปนี้:
news, sun, money, fx, chart, calendar, tasks, fuel, air, draft, image, file, audio, video, sparkles, heart, clock
(หากระบุชื่อไอคอนที่ระบบไม่รู้จัก จะแสดงผลเป็นรูป ⚡ ทันที)

🧩
คำเตือนเกี่ยวกับการใช้บล็อก html
ระบบจะเรนเดอร์บล็อก HTML ด้วยไลบรารีเนทีฟ (แปลง HTML เบื้องต้นให้เป็น Widget) — จะไม่มีการใช้ WebView ไม่มี JavaScript และไม่สามารถเขียน CSS Custom แทรกได้
คุณควรใช้ html เป็นทางเลือกสุดท้ายสำหรับจัดเลย์เอาต์พิเศษที่ไม่สามารถทำได้ด้วยบล็อกมาตรฐานเท่านั้น ไม่ใช่เพื่อเป้าหมายในการดัดแปลงสไตล์หรือหน้าตาของการ์ด

ตัวอย่างเคสใช้งานจริง

รายงานข่าวเศรษฐกิจ (News)

โครงสร้าง JSON ที่เซิร์ฟเวอร์ตอบกลับ:

{
  "carousel": [
    {
      "header": { "title": "กนง.ลดดอกเบี้ย เซอร์ไพรส์ตลาด" },
      "body": [
        {
          "type": "text",
          "text": "มีมติให้ปรับลดอัตราดอกเบี้ย 0.25% ครั้งแรกรอบปี หวังหนุนเศรษฐกิจ..."
        }
      ],
      "footer": {
        "icon": "news",
        "text": "สำนักข่าวเศรษฐกิจ",
        "trailing": "อ่านต่อ →",
        "action": { "data": "https://..." }
      }
    }
  ]
}

การแสดงผล (Rendering Result)

  • แสดงเป็นชุดการ์ดที่เลื่อนปัดไปมาได้ (Swipe) โดยมองเห็นใบถัดไปโผล่มาประมาณ 25% เพื่อบอกผู้ใช้ว่าเลื่อนต่อได้
  • Header: แสดงหัวข้อด้วยสีแบรนด์และตัวอักษรหนา
  • Body: เนื้อหาสรุปข่าว รองรับการเลื่อนอ่านภายในตัวการ์ดหากข้อความยาวเกินไป
  • Footer: แถบด้านล่างที่แตะสัมผัสได้ มุมซ้ายบอกแหล่งข่าว พร้อมมีคำว่า อ่านต่อ → ทางมุมขวา เมื่อกดแล้วจะทำการเปิดลิงก์เบราว์เซอร์

ตัวอย่างการ์ดสั้นแบบ Bignum (อากาศ)

{
  "header": { "icon": "sun", "title": "กรุงเทพมหานคร" },
  "body": [
    {
      "type": "bignum", 
      "value": "34°",
      "sub": "ต่ำสุด 27° · ฝน 40%"
    }
  ]
}
บทที่ 5

ส่งรีวิว & เผยแพร่

เส้นทางจากส่งผลงานไปจนถึงการปล่อย (Live) ให้ผู้ใช้งานจริงทุกคนสามารถดาวน์โหลดเครื่องมือของคุณได้

1. ส่งงานผ่านระบบ Portal

กระบวนการยื่นเรื่องให้ระบุ 3 อย่างหลัก ๆ ได้แก่: type (ประเภทของ Capability: tool, mcp, skill, subagent), name (ชื่อเครื่องมือ), และ payload (ข้อมูลการตั้งค่าแบบ JSON)

Typeสิ่งที่ต้องมีใน Payload
toolendpoint, description, parameters, argKeys และข้อมูลสำหรับโชว์หน้าร้าน (Display Info)
mcpurl, headers, และ Array ของ tools[] (ซึ่งทุกตัวต้องมี argKeys เช่นกัน)
skillinstructions (คำสั่ง Prompt) และค่า requires (หากมีการใช้เครื่องมือตัวอื่นช่วย)
subagentsystem (System Prompt), toolNames[] (เครื่องมือที่อนุญาตให้ใช้), และ maxSteps (จำนวนก้าวการทำงานสูงสุด)

2. Auto-audit (กระบวนการตรวจจับอัตโนมัติ)

ก่อนที่ใบคำขอของคุณจะถึงมือทีมงานปิ่น ระบบจะทำการตรวจสอบความปลอดภัยเบื้องต้น:

  • Block (ระงับ): หากพบ argKey ที่อยู่ในหมวดหมู่ต้องห้าม หรือเครื่องมือประเภท tool ไม่มีการระบุ Endpoint URL
  • Warn (แจ้งเตือน): หากไม่ประกาศ argKeys, เครื่องมือ skill ไม่มี Instructions, หรือ subagent ไม่ได้ระบุ System Prompt / รายชื่อเครื่องมือ

3. การพิจารณา (Review) และติดตั้ง

ผู้ดูแลระบบของปิ่นจะเข้ามาตรวจสอบคุณภาพ หากผ่านการอนุมัติ สถานะจะเปลี่ยนจาก pending ไปเป็น approved (อนุมัติ) และเครื่องมือของคุณจะถูกติดตั้งเข้าสู่ Catalog คุณสามารถติดตามสถานะทั้งหมดได้ตลอดเวลาในหน้า Portal

4. ข้อมูลสำหรับจัดแสดงในหน้าร้าน (Store Listing)

หากคุณระบุฟิลด์เหล่านี้ใน Payload เครื่องมือของคุณจะถูกนำเสนอในหน้าร้านเครื่องมืออย่างสวยงาม:

ฟิลด์คำอธิบาย
labelชื่อที่จะแสดงต่อผู้ใช้ทั่วไป (เช่น "เช็คพยากรณ์อากาศ")
blurbคำโปรยหรือสโลแกนสั้น ๆ ดึงดูดความสนใจ
category / providerหมวดหมู่การใช้งาน / ชื่อผู้ให้บริการหรือนักพัฒนา
pricingโมเดลราคา: free (ฟรี), onetime (จ่ายครั้งเดียว), subscription (จ่ายรายเดือน)
statusสถานะการให้บริการ: live (ใช้งานได้ทันที), trial (ทดลองใช้ฟรี), soon (เตรียมพบกันเร็ว ๆ นี้)
💡
ความสามารถเปิดโดยปริยาย (Opt-out):
ความสามารถที่อยู่ใน Catalog ปิ่นสามารถเรียกใช้ได้ทันทีโดยปริยาย ผู้ใช้ ไม่ต้องกดเปิดเอง — เช่น ปิ่นสามารถ "ดูดวง" ให้ได้ตั้งแต่ครั้งแรก ฟิลด์ status (เช่น trial) และ pricing ใช้สำหรับการแสดงผลและการคิดเงินในหน้าร้านเท่านั้น ไม่ได้ปิดกั้นการเรียกใช้ ผู้ใช้ที่ไม่ต้องการความสามารถใดสามารถปิด (Opt-out) ได้จากหน้า "ความสามารถ" ของแอป

5. การปรับค่า Params (หลังใช้งานจริง)

หลังจากที่เครื่องมือ Live แล้ว หากคุณมีความจำเป็นต้องปรับตั้งค่าเริ่มต้น (Defaults) เช่น เปลี่ยน RSS URL ต้นทางสำหรับระบบ News Reporter, หรือแก้ไขค่า Default Params ของเซิร์ฟเวอร์ MCP คุณสามารถปรับได้จากหน้า "การตั้งค่าเครื่องมือ" บนการ์ดใน Store ของคุณได้ทันที ค่าที่ปรับเปลี่ยนจะไหลไปถึงเครื่องลูกข่ายโดยอัตโนมัติ ไม่ต้องสั่ง Deploy แต่อย่างใด

6. การเผยแพร่ Catalog (Publishing) และเวอร์ชันอัปเดต

  • เมื่อเจ้าของกดยืนยัน "เผยแพร่ Catalog" ระบบจะทำการสร้าง Snapshot รุ่นใหม่พร้อมเซ็นรับรอง (Signed) โทรศัพท์มือถือของผู้ใช้จะดาวน์โหลด Snapshot ล่าสุดเพื่อใช้งานใน Runtime อย่างปลอดภัย
  • หากคุณทำการอัปเดต Schema ของระบบ MCP คุณสามารถสั่ง Refresh ผ่านหน้า Portal ได้ทันที โดยที่แอปพลิเคชันปิ่นตัวหลักของผู้ใช้งานไม่จำเป็นต้องถูก Rebuild หรือรออัปเดตเวอร์ชันจาก App Store เลย
